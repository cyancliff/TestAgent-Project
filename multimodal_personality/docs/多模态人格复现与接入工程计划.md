# 多模态人格复现与接入工程计划

## 1. 背景与目标

本计划面向论文《A multimodal personality prediction framework based on adaptive graph transformer network and multi-task learning》的工程复现与项目接入。

当前仓库 `TestAgent` 的核心能力是基于问卷与多智能体分析的 ATMR 心理测评，主链路围绕题目、作答、报告生成展开。论文方法则是基于视频、音频和转写文本进行 Big Five 人格预测，输入输出、依赖栈和训练流程均与现有系统不同。

因此，本项目采用以下策略：

1. 先将论文能力复现为独立的多模态人格预测子系统。
2. 再将该子系统以独立接口的形式接入现有 FastAPI 项目。
3. 初期只做结果展示与报告补充，不直接改写现有 ATMR 评分逻辑。

## 2. 本阶段范围

### 2.1 纳入范围

本阶段包含以下工作：

1. 设计独立的多模态人格模块目录结构。
2. 固化复现所需依赖与运行方式。
3. 搭建后端独立接口骨架。
4. 预留数据模型与报告接入方案。
5. 明确后续训练、推理、上线的迭代顺序。

### 2.2 暂不纳入范围

本阶段暂不处理以下事项：

1. 直接训练出可复现论文指标的模型。
2. 将 Big Five 分数与 ATMR 四维分数进行数值融合。
3. 改写现有问卷提交与评分链路。
4. 前端完整页面开发。

## 3. 总体架构

整体采用“双层结构”：

### 3.1 独立复现层

职责：

1. 管理论文相关依赖。
2. 处理视频、音频、文本预处理。
3. 负责特征提取、模型训练与推理。
4. 输出结构化 Big Five 结果。

建议目录：

```text
multimodal_personality/
  configs/
  data/
  checkpoints/
  preprocessing/
  feature_extractors/
  models/
  inference/
  evaluation/
  README.md
```

### 3.2 项目接入层

职责：

1. 接收上传的视频文件。
2. 调用独立复现层的推理入口。
3. 保存分析任务状态与结果。
4. 向报告系统提供 Big Five 结果。

建议目录：

```text
app/
  api/
    multimodal_personality.py
  services/
    multimodal_personality_service.py
  schemas/
    multimodal_personality.py
```

## 4. 复现策略

### 4.1 复现原则

论文正文与作者公开代码存在细节差异，因此复现策略采用“双轨对齐”：

1. 第一优先级：对齐作者公开代码，先保证可运行与可验证。
2. 第二优先级：对照论文结构，标记实现偏差并逐步修正。

### 4.2 最小可运行复现链路

第一阶段只要求实现以下能力：

1. 输入单个视频文件。
2. 抽取视频帧、音频和转写文本。
3. 构造四路输入特征：
   `clip_video`
   `wav2clip`
   `text`
   `scene-description/bg`
4. 调用模型前向推理。
5. 输出 Big Five 五维分数。

### 4.3 复现输出

推理结果至少包含以下字段：

```json
{
  "openness": 0.0,
  "conscientiousness": 0.0,
  "extraversion": 0.0,
  "agreeableness": 0.0,
  "neuroticism": 0.0,
  "model_version": "string",
  "status": "pending|running|completed|failed"
}
```

### 6.2 为什么不直接复用现有 `AssessmentSession`

原因如下：

1. 现有会话是问卷型测评，不包含多媒体资产生命周期。
2. 视频文件、推理任务、特征缓存属于新对象域。
3. 强行复用会提高模型和业务耦合度。

## 7. 依赖与环境策略

不建议将论文复现依赖直接并入主环境，原因如下：

1. 需要 `torch-geometric` 等较重依赖。
2. `wav2clip`、`ffmpeg`、旧版 `torch` 组合存在兼容性风险。
3. 主服务环境应保持稳定，避免因研究性依赖导致整体不可用。

因此建议：

1. 主项目保留现有依赖。
2. 多模态模块单列 `requirements_multimodal.txt`。
3. 训练环境与 API 推理环境后续可再细分。

## 8. 风险清单

### 8.1 数据风险

1. 若没有 `CFI-V2` 或 `UDIVA v0.5`，只能完成结构复现，不能完成论文指标复现。

### 8.2 环境风险

1. `torch-geometric` 与 CUDA/Windows 组合安装复杂。
2. 音视频预处理依赖系统级工具。

### 8.3 方法风险

1. 论文文字描述与公开代码实现并不完全一致。
2. 若直接按论文重写，结果可能与官方表现偏离。

### 8.4 业务风险

1. Big Five 与 ATMR 不是同一标签体系。
2. 早期不应直接做评分融合。

## 12. 接下来你要做什么

你当前最重要的任务不是改代码，而是把数据和运行条件整理到位。建议按以下顺序执行。

### 12.1 第一步：完成 CFI-V2 数据下载并落位

目标：

将数据放到统一目录，避免后续重复改路径。

建议目录：

```text
multimodal_personality/data/cfi_v2/
  train/
    videos/
    annotation.pkl
    transcription.pkl
  val/
    videos/
    annotation.pkl
    transcription.pkl
  test/
    videos/
    annotation.pkl
    transcription.pkl
  manifests/
```

如果你的实际目录结构和这个不一致，不需要强行重排，只需要保证你最终知道：

1. 训练集视频目录在哪里
2. 验证集视频目录在哪里
3. 测试集视频目录在哪里
4. 三个阶段的 annotation pickle 路径
5. 三个阶段的 transcription pickle 路径

### 12.2 第二步：生成 manifest

数据到位后执行：

```powershell
python scripts\build_cfi_v2_manifest.py --config multimodal_personality\configs\cfi_v2.example.yaml --phase all
```

如果你没有按模板路径放数据，就先改：

`multimodal_personality/configs/cfi_v2.example.yaml`

这一步完成后，应该生成：

1. `train_manifest.json`
2. `val_manifest.json`
3. `test_manifest.json`

位置默认在：

`multimodal_personality/data/cfi_v2/manifests/`

### 12.3 第三步：先做小样本预处理验证

不要一开始就跑 6000 条训练样本。先挑 5 到 20 个视频做试运行。

你需要确认：

1. 抽帧是否正常
2. 音频是否正常提取
3. transcript 是否非空
4. artifact 目录是否与视频一一对应

建议先用现有 API 或后续批处理脚本，验证少量样本通过后再扩大范围。

### 12.4 第四步：生成 CLIP 特征任务

manifest 生成后执行：

```powershell
python scripts\build_clip_feature_jobs.py --manifest multimodal_personality\data\cfi_v2\manifests\train_manifest.json
```

这一步会生成：

`multimodal_personality/data/cfi_v2/feature_jobs.json`

它的作用是把“哪些样本已经具备帧和 transcript”整理成后续特征提取任务。

### 12.5 第五步：先跑小批量 CLIP 特征提取

建议不要直接全量跑，先限制数量：

```powershell
python scripts\extract_clip_features.py --jobs multimodal_personality\data\cfi_v2\feature_jobs.json --output-dir multimodal_personality\data\cfi_v2\features\clip --limit 10
```

重点检查：

1. 模型权重是否能下载成功
2. 是否能生成每个样本的特征 JSON
3. 图像特征和文本特征维度是否稳定

### 12.6 第六步：把当前进度反馈给我

你完成上面步骤后，直接把以下信息给我：

1. 你的真实数据目录结构
2. manifest 命令输出
3. 是否存在缺视频或缺 transcript
4. CLIP 特征提取是否成功
5. 如果失败，贴最关键的一段报错

这样我就能继续接论文模型输入层，而不是重新做环境排查。
