## 1. 项目概况

- 项目名称：TestAgent / ATMR 智能心理测评系统
- 当前基线：`main@a628c0c`
- 当前版本：`pyproject.toml` 中定义为 `0.1.0`
- 产品目标：基于自适应选题、多智能体辩论和 RAG，输出 ATMR 四维心理测评报告，并提供测评后的 AI 咨询能力
- 当前状态：主流程已闭环，工程化基础已建立，开发重点已经从“继续堆功能”转向“稳定性、可维护性和部署一致性收口”

## 2. 当前系统快照

### 2.1 技术栈

| 层级 | 技术 |
|------|------|
| 后端 | FastAPI 0.104.1、Uvicorn 0.24、SQLAlchemy 2.0、Pydantic 2.5 |
| 前端 | Vue 3.5、Vite 8、Vue Router 4、Axios、Chart.js |
| AI / 检索 | AutoGen 0.2.14、LiteLLM、DeepSeek、Qwen、GLM、PageIndex |
| 数据层 | PostgreSQL 15（部署）、SQLite（开发模式） |
| 工程与运维 | Docker Compose、Nginx、slowapi、pytest、ruff、pre-commit |

### 2.2 关键目录与职责

| 路径 | 说明 |
|------|------|
| `app/main.py` | 应用入口，负责路由注册、限流、中间件、生命周期和健康检查 |
| `app/api/assessment/*` | 测评主链路，按会话、题目、提交、流式结果拆分 |
| `app/api/auth.py` / `app/api/chat.py` / `app/api/rag.py` | 认证、咨询对话、RAG 查询接口 |
| `app/api/multimodal_personality.py` | 可选扩展的多模态人格分析接口 |
| `app/services/*` | 自适应选题、异常检测、辩论编排、报告生成、阶段管理、题库清洗 |
| `app/models/*` + `app/core/database.py` | ORM、数据库连接与显式初始化 |
| `frontend/src/components/*` | 登录、答题、报告、聊天、历史记录和统一弹窗 |
| `scripts/*` | 题库导入、特征向量、迁移、监控与状态检查 |
| `tests/*` | 服务层与健康检查测试 |

### 2.3 当前主流程

```text
注册/登录
-> 创建测评会话
-> fixed 题
-> A / T / M / R 四个阶段答题
-> 每阶段触发模块辩论
-> 汇总生成综合报告
-> 历史记录查看
-> 基于报告继续聊天咨询
```

## 4. 关键实现记录

### 4.1 测评流程

- 测评主链路已经从早期单文件接口演进为 `sessions / questions / submissions / streaming` 四类职责拆分，维护成本明显降低。
- 答题编排采用 `intro / fixed / A / T / M / R` 阶段模型，`stage_service.py` 负责阶段推进，`scoring.py` 负责分值计算。
- 会话支持断点续答、阶段提交、已完成后修改答案和重新生成结果，前端对应页面为 `Assessment.vue`、`History.vue` 和 `Report.vue`。

### 4.2 自适应选题与异常检测

- `question_selection.py` 采用“贝叶斯能力估计 + Fisher 信息量 + 覆盖度 + 难度匹配 + 区分度”的混合选题策略。
- 能力估计以 `N(0.5, 0.25)` 为先验，随作答持续更新；异常作答会在估计过程中降权处理。
- `ai_detector.py` 已从单一时间阈值演进为风险评分模型，当前会综合作答过快、过慢、连续快速作答和重复选项等模式生成 `risk_score` 与追问文案。

### 4.3 多智能体辩论与 RAG

- 辩论引擎由 `debate_manager.py` 编排，角色分工为正方心理学专家、反方风控专家和裁决官，结果经 `streaming.py` 以 SSE 推送到前端。
- 报告生成分为“阶段辩论”和“综合报告”两层，先产出模块结论，再汇总生成完整心理画像。
- `rag_service.py` 已从关键词检索升级为“粗筛 + LLM 评分 + 内容提取”的三阶段流程，并新增超时重试与响应解析容错。

### 4.4 前端交互与展示

- 前端主页面包含 `Login.vue`、`Assessment.vue`、`Report.vue`、`Chat.vue`、`History.vue`，当前以响应式收口和状态一致性为重点。
- 辩论展示已从早期文本流改为轨道动画与阶段化加载反馈，降低等待过程中的信息噪声。
- `AppDialog.vue` 和 `useAppDialog.js` 统一了承载提示、确认和输入的弹窗模式，减少各页面重复维护弹窗状态的问题。

### 4.5 部署与运行

- 部署方案以 `docker-compose.yml` 为中心，组合 PostgreSQL、FastAPI 与前端 Nginx。
- 本地开发默认允许切换到 SQLite，部署环境以 PostgreSQL 为主；两种模式通过 `app/core/config.py` 统一配置。
- `init_db()` 当前仍承担显式建表和题库清洗职责，并在启动阶段通过 lifespan 调用。
- `scripts/import_data.py`、`question_sanitizer.py` 和 `tests/test_project_health.py` 组成了题库初始化和环境健康检查的基础闭环。

## 5. 关键设计决策

- 选择 FastAPI + Vue 分层，而不是单体模板渲染：后端专注 API 与 AI 编排，前端独立处理复杂交互、流式状态和图表展示。
- 选择多智能体辩论，而不是单模型直接出报告：报告需要有“支持观点 / 反方质疑 / 裁决汇总”的结构化推理链。
- 选择 RAG，而不是优先做模型微调：ATMR 知识库以结构化文档为主，检索增强更便于更新和追溯依据。
- 选择 SSE，而不是 WebSocket：当前核心场景是服务端向前端单向推送辩论进度，SSE 更轻量。
- 选择“SQLite 开发 + PostgreSQL 部署”双模式：优先降低本地启动成本，同时保留线上数据库能力。

## 6. 质量状态

- 已建立 `pytest`、`ruff` 和 `pre-commit` 基线，测试目录当前覆盖 `ai_detector`、`question_selection`、`rag_service`、`question_sanitizer` 和项目健康检查。
- 工程层面的收口重点已经从“继续加功能”转为“减少启动副作用、兼容多环境、增强容错和补足测试”。
- 与 4 月中旬之前相比，当前稳定性改进主要集中在四类问题：
  - 启动链路：lifecycle 初始化、显式建表、开发态 SQLite 兼容。
  - 数据质量：题库导入清洗、题干标准化、报告结构和时区一致性。
  - 交互一致性：统一弹窗、移动端布局、阶段加载反馈和聊天页面响应式。
  - 容错能力：RAG 超时重试、报告失败回退、头像路径修复、Docker 构建稳定性。

## 7. 当前风险与后续建议

- 数据库迁移体系仍未完全切到 Alembic 主导。当前 `init_db()` 仍会执行 `create_all`，更适合作为开发环境兜底，不适合作为长期生产迁移策略。
- 辩论和报告生成仍大量依赖线程与 SSE 协作，任务状态模型和失败重试机制还不够正式，后续应评估引入专门任务队列。
- Markdown 渲染安全策略尚未在所有页面统一，`Chat.vue`、`Report.vue`、`Assessment.vue` 的富文本输出仍值得继续收紧。
- 自动化测试目前以服务层和健康检查为主，API 级集成测试、报告生成失败路径和前端安全回归测试仍需补齐。
- 现有后续事项已经单独整理在 `docs/待完成任务.md`，建议把该文档视为实施清单，而本文作为维护者阅读入口。
