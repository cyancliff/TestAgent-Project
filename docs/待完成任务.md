# TestAgent 待完成任务

> 本文档基于 2026-04-15 对当前代码库的直接检查整理，不沿用旧文档结论。判断依据主要来自 `app/`、`frontend/src/`、`tests/` 和当前本地测试结果。

## 当前判断

项目已经有完整的主流程骨架：注册登录、测评答题、阶段提交、报告展示、聊天、RAG、历史记录都已落地。

真正需要优先处理的，不是再加新页面，而是把下面四类问题收紧：

1. 数据库初始化和迁移体系还不稳定。
2. 报告/辩论后台任务实现偏脆弱。
3. 前端存在富文本渲染安全风险。
4. 自动化测试目前既不完整，也不能稳定跑通。

---

## P0

### 1. 去掉导入即建表，正式切到 Alembic 迁移

**代码现状**

- `app/models/__init__.py` 在模块导入时直接执行 `Base.metadata.create_all(bind=engine)`。
- `alembic/` 已存在，但项目仍依赖自动建表。
- 当前执行 `python -m pytest -q` 会在测试收集阶段尝试连本地 PostgreSQL，并因为数据库未启动而失败。

**直接影响**

- 测试环境、脚本环境、导入工具时都有副作用。
- 数据库不可用时，连“导入模型”都会失败。
- 迁移历史不可控，线上结构演进风险大。

**建议动作**

- [ ] 生成并提交首个 Alembic migration。
- [ ] 删除 `app/models/__init__.py` 中的 `create_all`。
- [ ] 启动流程改成显式执行迁移，不再靠导入副作用。
- [ ] 测试环境使用独立测试库或 SQLite/临时 Postgres，而不是默认本地库。

### 2. 把线程式后台任务换成可追踪的任务系统

**代码现状**

- `app/api/assessment/streaming.py` 里的 `trigger_module_debate`、`generate_report_in_background` 依赖 `threading.Thread`。
- `app/services/debate_manager.py` 本身也在用线程 + queue 推消息。
- 任务状态没有独立表或统一状态机，只能通过 `report_content is None` 猜测“还在生成”。

**直接影响**

- 进程重启后任务状态丢失。
- 超时控制粗糙，线程超时后仍可能继续跑。
- 多实例部署时难以横向扩展。
- 前端只能轮询猜状态，无法可靠展示失败原因。

**建议动作**

- [ ] 引入 Celery / Dramatiq / ARQ 之一，替换线程式后台任务。
- [ ] 为“模块辩论”和“最终报告生成”建立任务状态字段。
- [ ] 保存开始时间、结束时间、失败原因、重试次数。
- [ ] 前端历史页和报告页改读明确状态，而不是依赖 `report_content` 是否为空。

### 3. 修掉前端 Markdown 渲染的 XSS 风险

**代码现状**

- `frontend/src/components/Chat.vue` 中 `formatMessage()` 直接 `marked.parse(...)` 后塞进 `v-html`。
- `frontend/src/components/Report.vue` 中 `renderMarkdown()` 同样直接 `marked.parse(...)` 后塞进 `v-html`。
- 当前没有看到 DOMPurify 或等价 HTML 清洗。

**直接影响**

- 只要模型输出、报告内容或数据库里混入恶意 HTML/脚本，前端就可能执行。
- 这类问题不属于“理论风险”，是典型的可利用面。

**建议动作**

- [ ] 引入 DOMPurify，对所有 `marked.parse()` 输出做 sanitize。
- [ ] 明确白名单标签和属性。
- [ ] 增加前端安全回归测试，至少覆盖 `<script>`、事件属性、`javascript:` URL。

### 4. 先把测试体系修到“可跑”，再谈扩展

**代码现状**

- 当前只有 `tests/test_ai_detector.py` 和 `tests/test_question_selection.py` 两组测试。
- 执行 `python -m pytest -q` 失败，原因是模型导入触发建表并强连本地 PostgreSQL。
- 还缺少认证、会话、阶段提交、报告接口、聊天接口的 API 级测试。

**直接影响**

- 改核心流程时没有回归保护。
- 当前连最基础的 CI 可运行性都不成立。

**建议动作**

- [ ] 先修复测试导入副作用，让 `pytest` 可在无本地数据库时运行。
- [ ] 补 FastAPI `TestClient` / `httpx.AsyncClient` 的接口测试。
- [ ] 补至少这几条主链路：登录、开始测评、保存答案、提交阶段、查看历史、查看报告、聊天。
- [ ] 把“报告生成失败”和“RAG 不可用”的降级场景也纳入测试。

## P1

### 5. 给报告生成建立显式状态模型

**代码现状**

- `app/api/assessment/streaming.py` 中历史接口把 `status == 'completed' && report_content is None` 视为“报告生成中”。
- 失败时直接把失败提示文案写进 `report_content`。

**问题**

- 数据字段职责混乱。
- 前端无法区分“生成中 / 生成失败 / 已完成但内容为空 / 内容异常”。

**建议动作**

- [ ] 在 `assessment_sessions` 中新增 `report_status`、`report_error`、`report_generated_at`。
- [ ] 失败信息单独存字段，不写进正式报告正文。
- [ ] 前端根据状态做展示，不再解析报告内容猜状态。

### 6. 优化历史页和报告接口的数据装载方式

**代码现状**

- `app/api/assessment/streaming.py` 的 `/history` 一次性拉用户所有已完成 session 的全部答题记录和题目。
- 其中存在按 session 反复 `any(...)` 匹配题目的逻辑，复杂度偏高。
- `/report/{session_id}` 一次返回完整报告、全部答题明细、分维度证据、模块辩论全文。

**问题**

- 数据量上来后，历史页和报告页接口会变重。
- 当前适合小规模数据，不适合长期累积。

**建议动作**

- [ ] 历史页只返回卡片必需字段，详情按需加载。
- [ ] 用更直接的 session_id / exam_no 分组映射，去掉多层 `any(...)` 扫描。
- [ ] 报告页拆分摘要接口与详情接口，避免一次返回过大载荷。

### 7. 收紧认证与上传校验

**代码现状**

- `app/api/auth.py` 有基础限流和头像大小/后缀限制。
- 但没看到用户名、密码强度、长度、字符集等明确约束。
- 头像上传主要校验扩展名和大小，没有做文件头校验。

**建议动作**

- [ ] 给用户名和密码加 Pydantic 约束。
- [ ] 对头像做 MIME/文件头双重校验。
- [ ] 为登录失败、注册失败、头像上传失败增加审计日志。

### 8. 补运行态监控，而不只是 `/health`

**代码现状**

- `app/main.py` 只有基础 `/health`。
- 日志还是 `logging.basicConfig(...)` 文本输出。

**建议动作**

- [ ] 增加结构化日志。
- [ ] 采集报告生成耗时、失败率、RAG 命中率、聊天调用耗时。
- [ ] `/health` 至少区分应用状态、数据库状态、关键依赖状态。

### 9. 清理配置层的旧写法和默认风险

**代码现状**

- `python -m pytest -q` 期间出现 `PydanticDeprecatedSince20`，说明配置层仍在使用旧版 class-based config。
- 同时出现 `SECRET_KEY` 默认值告警。

**建议动作**

- [ ] 把 `app/core/config.py` 迁到 Pydantic v2 推荐写法。
- [ ] 启动时把生产环境弱 `SECRET_KEY` 视为硬错误，而不是仅 warning。
