### 第一部分：系统需求分析

需求分析解决的是“系统必须做什么”的问题。根据毕业设计任务书，本系统是一个融合传统 ATMR 评分机制与大模型个性化解释的新型心智测评系统。

**1. 业务流程需求 (核心痛点解决)**

- **传统痛点突破**：传统静态报告缺乏个性化，且忽略了用户的“作答过程行为”（如作答时长、犹豫程度）。本系统必须在答题过程中实时捕捉这些隐式数据。
    
- **快车道（实时拦截）**：用户回答问题后，系统需通过轻量化 AI 瞬间判断其是否存在异常行为（如“乱猜”、“误导”或“过度自信”），若有异常则动态生成追问。
    
- **慢车道（异步辩论）**：ATMR 测评包含认知能力、情感管理、行为模式、发展潜力四个核心维度。每完成一个维度的测评，系统需在后台异步唤醒多智能体（正方、反方、裁决 Agent）开展多轮自动辩论。
    

**2. 系统功能需求**

- **测评库管理模块**：支持 ATMR 心智测评题库的导入、编辑与管理，支持单选、多选及情景题等灵活配置。
    
- **智能诊断与报告生成模块**：
    
    - 依托 ATMR 标准计分规则生成基础评分结果。
        
    - 构建轻量级解释知识库，利用大模型提取内容组成证据链与推理链。
        
- **安全与运维保障模块**：实现用户管理与权限控制、日志记录与审计、模型调用监控与异常报警机制，并对测评数据进行加密存储。

### 第二部分：系统概要设计 

概要设计解决的是“系统怎么划分模块和通信”的问题。本系统采用典型的前后端分离及微服务化思想，分为表现层、业务逻辑层和 AI 算法层。

**1. 表现层 (Web 前端 - 基于 Vue.js)**

- **动态问答组件**：负责渲染 ATMR 题目，丝滑切换题目，并在用户答题过快时无缝弹出轻量化 AI 的“追问对话框”。
    
- **行为数据采集器**：静默记录用户在每道题上的停留时间、选项修改轨迹，随答题结果一并封装为 JSON 发送给后端。
    
- **可视化报告渲染大屏**：通过 WebSocket 或轮询机制拉取后台辩论进度。测试完成后，渲染包含 ATMR 四维度雷达图、文字解析及个性化的深度诊断报告。
    

**2. 业务逻辑层 (后端 - 基于 Spring Boot)**

- **API 网关与鉴权**：负责拦截所有前端请求，验证用户登录状态（Token），实现权限控制。
    
- **测评调度中心（核心）**：
    
    - 接收前端的答题数据，转发给 Python 算法层进行异常检测。
        
    - 监听当前用户的测评进度。当用户完成 ATMR 某一维度的所有题目时，开启**异步线程 (`@Async`)**，打包该维度数据并调用 Python 端的多智能体辩论接口。
        
- **数据持久化服务**：通过 MyBatis/JPA 与 MySQL 数据库交互，对用户的测评过程数据和最终结果进行加密存储。
    

**3. AI 算法层 (核心引擎 - 基于 Python + AutoGen + DeepSeek)** 本层作为独立的服务运行，专职处理高密度的逻辑推理，主要分为两个子模块：

- **实时检测引擎 (Detector Agent)**：一个极轻量级的 Agent，嵌入了认知诊断模型（CDM）的数值判定逻辑。瞬间分析前端传来的行为数据，输出是否需要“生成追问”的指令。
    
- **多智能体辩论框架**：基于 AutoGen 构建。
    
    - **正方 Agent**：挖掘该维度下的积极特征。
        
    - **反方 Agent**：挑刺并提示潜在心理风险。
        
    - **裁决 Agent**：综合各方意见，结合 RAG（检索增强生成）从本地 ATMR 知识库中提取支撑论据，输出包含明确证据链的该维度总结报告。`*
